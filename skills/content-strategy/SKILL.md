# Content Strategy — 컨텐츠 전략 수립 스킬

이력서와 공고 요구사항을 매핑하여 구체적인 수정 전략을 수립하는 스킬.

## 실행 조건
- 이력서 데이터와 공고 분석 결과가 모두 준비된 경우
- 시나리오 A(맞춤수정)로 진행 중인 경우
- 피드백 기반 수정 전략 업데이트가 필요한 경우
- `experience_blueprint` 모드에서 공고 스택 기반 프로젝트 기획 방향이 필요한 경우

## 워크플로우
### Step 1: 기술 스택 매칭 분석
1. 이력서의 기술 스택과 공고 필수/우대 기술 비교
2. 매칭되는 기술, 누락된 기술, 약한 부분 식별
3. 각 기술의 중요도 등급 매기기 (Critical/Important/Nice-to-have)

### Step 2: 적합도(Fit Score) 산정 — 별 10점 만점

공고별로 다음 5개 항목을 각각 0-10점으로 평가하고, 별점으로 시각화하라:

| 평가 항목 | 점수 기준 | 가중치 |
|----------|---------|-------|
| **기술 스택 매칭** | 필수 기술 보유율 × 10, 우대 기술은 0.5배 가중 | 35% |
| **경험 연관성** | 프로젝트/경력이 공고 직무와 얼마나 관련있는지 | 30% |
| **경력 수준 적합성** | 요구 경력 대비 초과/미달 정도 (적정 = 높은 점수) | 15% |
| **키워드 밀도** | 이력서 내 공고 핵심 키워드 포함률 | 10% |
| **산업/도메인 경험** | 같은 업종/도메인 경험 보유 여부 | 10% |

**별점 시각화 규칙:**
- 9.0~10.0: ★★★★★ (완벽한 매치)
- 7.0~8.9: ★★★★☆ (강한 매치, 약간의 보완 필요)
- 5.0~6.9: ★★★☆☆ (보통, 개선 여지 있음)
- 3.0~4.9: ★★☆☆☆ (약한 매치, 많은 개선 필요)
- 0.0~2.9: ★☆☆☆☆ (미달, 지원 재고 권장)

각 항목의 점수와 근거를 상세히 기록하고, 종합 점수도 계산하라.

### Step 3: 프로젝트 경험 재평가
1. 각 프로젝트를 공고 요구사항과의 연관성으로 재평가
2. 연관성이 높은 프로젝트는 상세히, 낮은 프로젝트는 간략히
3. 누락된 경험이 있다면 (있는데 이력서에 안 쓴 경우) 추가 제안

### Step 4: 섹션별 수정 전략 수립
각 섹션에 대해:
- `action`: `keep` / `modify` / `expand` / `condense` / `remove`
- `reason`: 변경 이유 (공고의 어떤 요구사항 때문인지)
- `suggested_focus`: 강조해야 할 포인트
- `suggested_keywords`: 포함해야 할 ATS 키워드

### Step 5: Fit Score JSON 출력
`_workspace/02_fit_score.json`에 fit 점수와 항목별 근거 저장.
`_workspace/02_strategy.json`에 섹션별 전략 저장.

### Step 6: Experience Blueprint 전략 (필요 시)
`_workspace/01_scenario.json`의 `work_mode`가 `"experience_blueprint"`이면:

1. 공고의 필수/우대 스택을 추출하고 Critical/Important/Nice-to-have로 분류한다.
2. 프로젝트 기획에서 사용할 공고 스택 매핑을 만든다.
3. 서비스 도메인에서 발생할 수 있는 실제 문제 상황을 정의한다.
4. 문제 해결 방향을 데이터 흐름, 상태 관리, 캐싱, 권한, 관측, 테스트 같은 구현 단위로 구체화한다.
5. 완성 후 이력서 bullet에 사용할 수 있는 측정 지표 후보를 정의한다.
6. 프로젝트명과 slug를 정하고, 4개 기획서가 각각 별도 md 파일로 생성되도록 파일 경로를 제안한다.

## 원칙
- 변경 이유를 항상 공고의 특정 요구사항과 연결하라 — "공고에서 '팀 리더십'을 요구하므로 리드 경험을 강조"처럼
- 보수적으로 접근하라 — 불필요한 변경은 피하고, 변경이 확실히 도움이 되는 부분만 수정 대상으로 삼아라
- 정량적 비교를 하라 — 기술 매칭률, 경험 연관성 점수 등 객관적 지표를 포함하라
- 문제해결 프로젝트는 공고의 필수/우대 스택을 우선 사용하라
- 문제해결 프로젝트 기획서는 하나로 합치지 말고, 4개 프로젝트별 md 파일과 JSON 인덱스로 분리하라
- 완료 전 프로젝트를 완료 경험처럼 쓰지 말고, 완성 후 사용할 수 있는 bullet로 분리하라

## 검증
- [ ] 모든 섹션에 대한 action이 정의되었는가?
- [ ] 각 변경에 공고 기반의 reason이 있는가?
- [ ] ATS 키워드가 전략에 포함되었는가?
- [ ] 우선순위가 매겨졌는가?
- [ ] 5개 Fit Score 항목이 모두 계산되었는가?
- [ ] 각 점수에 구체적 근거가 포함되었는가?
- [ ] 별점 시각화가 올바른가?
