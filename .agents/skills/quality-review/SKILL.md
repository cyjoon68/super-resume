# Quality Review — 이력서 품질 검증 스킬

이력서의 언어적, 내용적, 기술적 품질을 종합적으로 검증하는 스킬.

## 실행 조건
- 수정된 이력서 초안이 준비된 경우 (시나리오 A/D)
- 사용자가 이력서만 제공하고 피드백을 요청한 경우 (시나리오 B)
- 최종 출력 전 품질 게이트가 필요한 경우

## 워크플로우
### Step 1: 언어 품질 검사
다음 항목을 검사하고 오류를 P0-P2로 등급 분류:
- 맞춤법 오류
- 문법 오류 (주어-동사 불일치, 시제 불일치)
- 띄어쓰기 오류
- 기술 용어 대소문자 오류 (예: "Github" → "GitHub")
- 중복 표현, 불필요한 워딩

### Step 2: 말투 및 어조 분석
- `_workspace/01_scenario.json`의 `tone` 값을 읽어서 **선택된 말투와 실제 작성된 말투가 일치하는지** 검증
- 전반적인 어조 평가 (전문적인가? 자신감 있는가? 간결한가? 기술적인가? 협업 중심인가?)
- 선택된 말투의 규칙을 위반한 표현 식별 (예: `concise` 말투인데 불필요하게 긴 문장, `confident` 말투인데 수동적인 표현)
- 수동태 과다 사용 여부
- 너무 수동적이거나 공격적인 표현 식별
- 업계/직군에 적합한 어조인지 평가
- 말투 일관성 점수 산정 (1-10): 전체 이력서가 동일한 말투로 유지되는지
- 말투 불일치가 발견되면 P1 등급으로 기록

### Step 3: 내용 일관성 검증
- 날짜/기간의 논리적 일관성 (겹치거나 비지 않는 기간)
- 역할과 경력 수준의 일관성
- 기술 스택과 프로젝트 경험의 일관성 (Java 프로젝트인데 기술 스택에 Java가 없는 경우)
- 이력서 전체에 걸쳐 동일 용어 사용 (예: "Frontend"와 "Front-end" 혼용)

### Step 4: ATS 호환성 검토
1. 공고 키워드와 이력서 키워드 매칭률 계산
2. 중요한 누락 키워드 식별
3. 섹션 헤더가 ATS 친화적인지 확인
4. 복잡한 형식(표, 이미지)이 ATS 파싱을 방해하지 않는지 확인

### Step 5: 종합 평가 리포트 생성
1. `_workspace/04_review_report.json`에 저장
2. 리포트 항목:
   - `overall_score`: 1-10 점수
   - `issues`: 발견된 문제 (등급별 분류)
   - `strengths`: 강점
   - `suggestions`: 개선 제안
   - `ats_match_rate`: ATS 키워드 매칭률 (%)
   - `summary`: 종합 평가 요약 (3-5문장)
3. P0 이슈가 있으면 자동 수정 → `_workspace/04_corrected_content.md` 저장
4. P0 이슈가 없으면 원본 유지

## 원칙
- 객관적 기준으로 평가하라 — "좋다/나쁘다"가 아니라 구체적 근거와 함께 평가하라
- P0 이슈(오타, 사실 오류)는 자동 수정하되 변경사항을 투명하게 공개하라
- P1-P2 이슈는 추천으로 제시하고, 사용자에게 수정 여부를 선택하게 하라
- ATS 매칭률은 맹신하지 말고, 키워드의 자연스러운 배치를 더 중요하게 평가하라

## 검증
- [ ] 모든 오류가 등급(P0/P1/P2)과 함께 기록되었는가?
- [ ] P0 이슈가 자동 수정되었는가?
- [ ] ATS 매칭률이 계산되었는가?
- [ ] 종합 점수와 평가 요약이 포함되었는가?
- [ ] 시나리오 B의 경우, 피드백 리포트가 실행 가능한 조언을 포함하는가?
