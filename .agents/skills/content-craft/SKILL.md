# Content Craft — 이력서 컨텐츠 작성 스킬

전략에 기반하여 이력서 내용을 실제로 작성하고 수정하는 스킬.
사용자가 선택한 **말투(tone)** 에 따라 표현 방식이 달라진다.

## 실행 조건
- 수정 전략(`02_strategy.json`)이 준비된 경우
- 시나리오 A/B에서 내용 수정이 필요한 경우
- 중간 버전을 생성하여 사용자 피드백을 받아야 하는 경우

## 말투(Tone) 옵션

이 스킬은 `_workspace/01_scenario.json`의 `tone` 값을 읽어 적용한다.
값이 없으면 `"preserve"`(말투유지)를 기본값으로 처리한다.

### 1. professional (전문적인)
**격식 있고 정확한 표현, 표준 업계 용어 사용. 가장 무난하고 보편적인 말투.**

| 항목 | 규칙 |
|------|------|
| 문장 구조 | 완전한 문장, 적절한 형용사/부사 사용 |
| 동사 | "개발했다", "설계했다", "구축했다", "담당했다" |
| 어조 | 중립적이고 객관적, 약간의 형식감 유지 |
| 길이 | 각 항목 1-3문장, 적당한 상세도 |
| 예시 | "마이크로서비스 아키텍처 기반의 결제 시스템을 설계하고 개발했다" |

### 2. confident (자신감 있는)
**강한 액션 동사, 성과 수치 전면 배치, 주도적이고 임팩트 있는 표현.**

| 항목 | 규칙 |
|------|------|
| 문장 구조 | 간결하고 강한 문장, 수식어 최소화 |
| 동사 | "주도했다", "최적화했다", "혁신했다", "구축했다" (가장 강한 동사 우선) |
| 어조 | 자신감 있고 적극적, 성과를 전면에 내세움 |
| 길이 | 각 항목 1-2문장, 핵심 성과부터先说 |
| 예시 | "결제 시스템 전면 재구축: 트랜잭션 처리량 300% 향상, 장애율 0.01%로 감소" |

### 3. concise (간결한)
**핵심만 간략하게, 불필요한 수식어와 부연 설명 제거. bullet point 위주.**

| 항목 | 규칙 |
|------|------|
| 문장 구조 | 구/절 단위, 완전한 문장보다 키워드 중심 |
| 동사 | 생략 가능 ("개발", "설계", "최적화" — 동사 없이 명사형) |
| 어조 | 직설적, 사실만 전달 |
| 길이 | 각 항목 10-15단어 이내 |
| 예시 | "결제 시스템 설계 · 개발 — 마이크로서비스 아키텍처, 월 50만 건 처리" |

### 4. technical (기술 중심)
**기술적 세부사항과 아키텍처 결정을 강조. 성능 수치, 기술 스택, 설계 패턴 중심.**

| 항목 | 규칙 |
|------|------|
| 문장 구조 | 기술적 맥락을 포함한 완전한 문장 |
| 동사 | "설계했다", "최적화했다", "도입했다", "마이그레이션했다" |
| 어조 | 엔지니어링 중심, 구체적인 기술 용어 사용 |
| 길이 | 각 항목 2-4문장, 기술적 설명 포함 |
| 예시 | "Spring Cloud Gateway 기반 API Gateway 도입으로 라우팅 지연 40ms→8ms 단축. Cassandra 클러스터 노드 3→7로 확장하여 읽기 처리량 280% 향상" |

### 5. collaborative (협업 중심)
**팀워크, 리더십, 크로스펑셔널 협업, 멘토링, 커뮤니케이션 능력 강조.**

| 항목 | 규칙 |
|------|------|
| 문장 구조 | 팀/조직 맥락을 포함한 완전한 문장 |
| 동사 | "협업했다", "리드했다", "조율했다", "멘토링했다", "가이드했다" |
| 어조 | 포용적이고 협력적, "팀", "함께", "조직" 키워드 자주 사용 |
| 길이 | 각 항목 2-3문장, 협업 맥락과 결과 포함 |
| 예시 | "4명의 개발자로 구성된 팀을 리드하여 결제 시스템 마이크로서비스 전환 프로젝트를 진행했다. 주 1회 스프린트 리뷰와 페어 프로그래밍 세션을 도입해 팀 생산성을 40% 향상시켰다" |

### 6. preserve (말투유지) — 기본값
**사용자의 원본 이력서 말투를 분석하여 그대로 유지. 개성과 스타일을 보존하면서 내용만 최적화.**
> 말투유지가 기본값이다. 이력서 원본이 없으면(GitHub만 제공) professional로 폴백된다.

| 항목 | 규칙 |
|------|------|
| 동작 방식 | `_workspace/01_tone_profile.json`의 분석 결과를 기반으로 원본 말투를 모방 |
| 분석 대상 | 문장 길이, 동사 선택 패턴, 형용사 사용 빈도, 수동태/능동태 비율, 전문 용어 사용 수준 |
| 어조 | 원본 그대로 — 사용자의 자연스러운 글쓰기 스타일을 유지 |
| 주의 | `_workspace/01_tone_profile.json`이 없으면 `"professional"`로 폴백 |

## 워크플로우
### Step 0: 말투(Tone) 읽기
1. `_workspace/01_scenario.json`에서 `tone` 필드 읽기
2. 값이 없거나 비어있으면 `"preserve"`로 간주
3. tone 값을 모든 작성 단계에서 참조

### Step 0-B: 말투유지(preserve) 처리 — 원본 말투 분석 및 모방
tone이 `"preserve"`이면 아래 프로세스를 실행한다:

#### 1. 말투 프로필 로드
- `_workspace/01_tone_profile.json`이 있는지 확인
- 없으면 `"professional"`로 폴백 (이력서 원본이 없는 상태)
- 있으면 말투 프로필의 특징을 추출

#### 2. 말투 특징 추출 (다음 항목을 원본 이력서에서 분석)
원본 이력서(`_workspace/01_original_resume.md`)에서 다음 특징을 추출:
- **평균 문장 길이**: 짧은 문장(10단어 미만) / 중간(10-20) / 긴(20+)
- **동사 스타일**: 능동태 비율, 선호하는 동사 유형 ("했다"형 vs "함"형)
- **형용사/부사 사용 빈도**: 절제된 vs 풍부한
- **기술 용어 밀도**: 높음(모든 문장에 기술 용어) / 중간 / 낮음(일상어 위주)
- **bullet point vs 문장형**: bullet point 위주 vs 완전한 문장 vs 혼합
- **수치 사용 빈도**: 자주 사용 vs 가끔 사용 vs 거의 없음
- **1인칭 사용**: "나는", "제가" vs 무인칭
- **전반적 톤**: formal / casual / technical / passionate / humble 등

#### 3. 말투 모방 규칙
1. 원본 이력서의 문장 길이 분포를 유지하라
2. 원본에서 주로 사용된 동사와 표현을 우선 사용하라
3. 원본의 기술 용어 밀도 수준을 유지하라
4. 원본의 형식 수준을 유지하라 (격식체 → 격식체, 반말체 → 반말체)
5. 새로운 내용을 추가할 때도 원본 말투와 일관성을 유지하라
6. **원본의 오타나 비문법적 표현은 교정하되, 말투 특성(짧은 문장, 특정 동사 선호 등)은 유지하라**
7. 말투 프로필을 `_workspace/03_tone_applied.json`에 저장

### Step 1: 원본 복사 및 전략 적용
1. `_workspace/01_scenario.json`의 `output_target`을 확인한다.
   - `resume`: 이력서만 작성/수정
   - `portfolio`: 포트폴리오만 작성/수정
   - `both`: 이력서와 포트폴리오를 모두 작성/수정
2. 원본 이력서 또는 포트폴리오 구조를 복사한다. 입력 자료와 산출물은 독립적으로 다룬다.
3. `candidate_notes.military_service`가 있으면 기본 정보, 지원 조건, 공고 적합성 섹션 중 자연스러운 위치에 반영한다.
4. 전략의 각 섹션별 action에 따라 처리:
   - `keep`: 원본 유지
   - `modify`: 말투에 맞춰 표현 개선
   - `expand`: 말투에 맞춰 내용 추가
   - `condense`: 간결한 말투면 더 축약, 아니라도 불필요한 부분 제거
   - `remove`: 제거

### Step 2: 프로젝트 설명 재구성 (STAR 형식)
각 프로젝트를 STAR 형식으로 재구성하되, **말투에 따라 표현 방식을 조정:**
- **professional**: "Situation → Task → Action → Result" 순차적으로 서술
- **confident**: "Result → Action" 순서로 결과를 먼저 강조
- **concise**: "Action → Result"만 간략히, 핵심 키워드 위주
- **technical**: "Situation(기술적 맥락) → Task → Action(기술적 세부사항) → Result(성능 수치)"
- **collaborative**: "Situation(팀 맥락) → Task(팀에서의 역할) → Action(협업 방식) → Result(팀 성과)"

### Step 3: 키워드 최적화
1. 전략에서 식별된 핵심 키워드를 자연스럽게 분산 배치
2. 키워드 과다 사용(stuffing) 방지
3. 기술 용어의 정확한 표기법 확인 (예: Node.js, React, TypeScript)
4. 말투에 따라 키워드를 배치하는 방식 차별화:
   - **professional**: 문장 속에 자연스럽게 녹임
   - **confident**: 목차/헤더에서 강조
   - **concise**: bullet point 앞부분에 배치
   - **technical**: 기술 스택 섹션과 경험 설명에 중복 배치
   - **collaborative**: "팀 내에서 React 기반 ..." 식으로 맥락과 함께

### Step 4: 중간 버전 스냅샷 저장
1. `output_target`에 따라 저장한다.
   - 이력서: `_workspace/03_draft_resume_v{N}.md`
   - 포트폴리오: `_workspace/03_draft_portfolio_v{N}.md`
   - 둘 다: 두 파일 모두 저장
2. `_workspace/03_changelog_v{N}.json`에 변경사항 요약 저장
   - 적용된 말투(tone)도 함께 기록
3. 변경사항 요약에는 다음 포함:
   - 변경된 섹션 목록
   - 각 변경의 이유
   - 이전 버전과의 diff 포인트

## 원칙 (말투 공통)
- "보여주라 말하지 마라" — "팀워크가 좋다"라고 쓰지 말고, "3명의 팀에서 크로스펑셔널 협업으로 프로젝트를 기한 내 배포했다"처럼 증거를 제시하라
- 능동태를 사용하라 — 선택한 말투와 관계없이 "개발했다", "설계했다" (수동태 "개발되었다" ❌)
- 구체적인 숫자를 포함하라 — "사용자 5만 명", "응답 시간 60% 개선"
- 모든 중간 버전은 보존하라 — 사용자가 이전 버전으로 돌아가길 원할 수 있다
- 버전 N을 생성한 후에는 사용자에게 보여주고 피드백을 요청하라
- **선택한 말투를 일관성 있게 유지하라** — 이력서 전체가 같은 말투로 작성되어야 한다. 섹션마다 다른 말투가 섞이지 않게 주의하라
- **말투가 내용을 왜곡하지 않게 하라** — confident 말투라고 실제보다 과장된 경험을 쓰지 마라
- **완료 전 프로젝트를 완료 경험처럼 쓰지 마라** — experience blueprint에서 나온 문장은 실제 구현/측정 전에는 "완성 후 사용할 수 있는 bullet"로만 제공하라
- **지원 조건을 누락하지 마라** — `candidate_notes.military_service`에 `산업기능요원 보충역` 같은 값이 있으면 최종 산출물에 유지하라
